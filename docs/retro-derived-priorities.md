# Retro-derived priorities

The SG Retro and Archive board was useful because it confirms several recurring pains that are directly relevant to tooling.

## Repeated or explicit signals

- "We need an actual 3D Car Integration strategy (mostly about testing)"
- "Too many obvious findings during the internal rack sessions"
- "Some procedures are a bit confusing and not documented"
- "Ticket definition and order / dependencies should be better documented"
- "Finding out car specific configurations is nigh impossible"
- "Unity build process and CarLib integration is an annoying black box"
- "Multiple parts need to be built and then moved manually; hard to debug"

## What this PoC should target first

1. Catch deterministic issues before rack time.
2. Produce reusable evidence instead of vague review comments.
3. Reduce obvious findings by enforcing preflight.
4. Normalize repeated checks into one consistent CLI + report flow.

## What it does not solve yet

- design-process alignment
- BMW communication lag
- ownership ambiguity
- meeting / role / onboarding politics
